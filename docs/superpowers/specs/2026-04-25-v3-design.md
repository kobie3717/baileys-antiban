# baileys-antiban v3.0 Design Spec

**Date:** 2026-04-25  
**Author:** Friday  
**Status:** Approved

---

## Goal

Ship v3.0 — zero-config DX + real ban protection improvements. Breaking changes acceptable (43 stars, low migration cost).

Two outcomes:
1. `new AntiBan()` works in 30 seconds with no config reading
2. Better ban protection: group profiles, state persistence, health recovery

---

## Public API Changes

### Constructor (breaking change)

```ts
// v2.x (deprecated)
new AntiBan({ rateLimiter: { maxPerMinute: 8 }, warmUp: { warmUpDays: 7 }, ... })

// v3.0
new AntiBan()                    // zero config — conservative preset
new AntiBan('moderate')          // preset string
new AntiBan('aggressive')        // preset string
new AntiBan({
  preset: 'moderate',            // base preset (optional, default: 'conservative')
  persist: './state.json',       // state survives restarts
  groupProfiles: true,           // stricter limits for @g.us JIDs
  maxPerMinute: 12,              // override any single preset value
  maxPerHour: 400,
  warmupDays: 5,
  logging: false,
})
```

### Preset definitions

| param | conservative | moderate | aggressive |
|---|---|---|---|
| maxPerMinute | 5 | 10 | 20 |
| maxPerHour | 100 | 300 | 800 |
| maxPerDay | 800 | 1500 | 4000 |
| minDelayMs | 2500 | 1500 | 800 |
| maxDelayMs | 7000 | 5000 | 3000 |
| warmupDays | 10 | 7 | 4 |
| day1Limit | 15 | 20 | 35 |
| autoPauseAt | 'medium' | 'high' | 'critical' |
| groupMultiplier | 0.5 | 0.7 | 0.9 |

### Method surface — unchanged

`beforeSend(jid, text)`, `afterSend(jid, text)`, `onDisconnect(reason)`, `onReconnect()`, `onIncomingMessage(jid, text)`, `getStats()`, `pause()`, `resume()`, `reset()`, `destroy()`

## New Files

### `src/presets.ts`
- Exports `PRESETS` record and `resolveConfig(input)` function
- `resolveConfig` accepts `string | AntiBanConfig | undefined` → normalized flat config
- Merges overrides on top of preset base

### `src/persist.ts`
- `StateManager` class
- `load(path): PersistedState | null` — wraps `JSON.parse` in try/catch, returns null on any error (corrupt/missing/bad schema), logs `[baileys-antiban] WARN: corrupt state file, starting fresh`
- `save(path, state): void` — debounced 5s write via `setTimeout`
- Save triggers: `afterSend()` AND new `afterHealthEvent()` — ensures ban/restriction events persist even if no send follows
- `PersistedState` shape: `{ warmup: WarmUpState, knownChats: string[], savedAt: number, version: 3 }`
- `knownChats` capped at 1000 entries (LRU eviction — drop oldest). Prevents unbounded growth on high-volume bots.
- **Single-writer assumption:** no file lock. Document as limitation — two processes sharing same state.json = race condition on concurrent writes. Use separate state files per process.

### `src/profiles.ts`
- `isGroup(jid: string): boolean` — `jid.endsWith('@g.us')`
- `isNewsletter(jid: string): boolean` — `jid.endsWith('@newsletter')` — v3 aliases to group behavior (same multiplier). Noted for v4 separate profile.
- `applyGroupMultiplier(limits, multiplier): limits` — scales perMinute/perHour/perDay
- Used in `antiban.ts` `beforeSend()`: if `groupProfiles` enabled and `isGroup(jid) || isNewsletter(jid)`, pass scaled limits to rateLimiter check
- v4 backlog: `groupOverrides: Record<jid, Partial<RateLimitConfig>>` for per-group limit overrides

### `src/cli.ts`
- Registered as `bin.baileys-antiban` in package.json
- Commands:
  - `status [--state <path>] [--json]` — print warmup day, today limits, health score. `--json` outputs machine-readable JSON for scripting/monitoring integrations
  - `reset [--state <path>]` — delete state file, confirm prompt
  - `warmup --simulate <days> [--preset <name>]` — print daily limit schedule with real calendar dates (e.g. `Mon Apr 28: 20 msgs/day`) not just counts
- No Baileys connection needed — works offline on state files

## Modified Files

### `src/antiban.ts`
- Constructor: accept `string | AntiBanConfig | undefined`
- Call `resolveConfig()` from presets.ts to get flat config
- On construction: if `persist` set, load state via `StateManager`
- In `afterSend()`: if persist set, call `stateManager.save()` (debounced)
- Pass `groupProfiles` flag through to `beforeSend()` logic
- Compat shim: detect old nested shape (`config.rateLimiter?.maxPerMinute` exists), map to flat, `console.warn` once (WARN level — visible in prod logs)

### `src/health.ts`
- Add `lastBadEventTime: number` field — updated ONLY on bad events (`recordDisconnect`, `recordMessageFailed`, `recordReachoutTimelock`). NOT on `recordReconnect()`. Active bots sending normally do not reset the decay clock.
- In `getStatus()`: after computing raw score, apply tiered decay:
  ```ts
  const minutesSinceLastBad = (now - this.lastBadEventTime) / 60000
  // Slow decay for high-severity events (403/401), fast for low-severity (disconnects)
  const decayRate = lastEventWasSevere ? 2 : 5  // pts/min
  score = Math.max(0, score - Math.floor(minutesSinceLastBad * decayRate))
  ```
- HIGH severity (403 forbidden, 401 logged out): 2pts/min decay → ~50 min to clear
- LOW severity (disconnect, message fail): 5pts/min decay → ~20 min to clear
- Add `lastEventWasSevere: boolean` field, set true on 403/401 events

### `src/warmup.ts`
- Remove `statePath` from `WarmUpConfig` (dead code)
- Persistence now handled by `StateManager` in `antiban.ts`
- No other changes

### `package.json`
- Add `bin: { "baileys-antiban": "./dist/cli.js" }`
- Bump version to 3.0.0

## Data Flow

```
new AntiBan('moderate', { persist: './state.json', groupProfiles: true })
  → resolveConfig() → flat merged config
  → StateManager.load('./state.json') → restore warmup state + knownChats
  → construct all modules with resolved config

beforeSend(jid, text)
  → isGroup(jid) + groupProfiles? → scale rate limits
  → health.isPaused()?
  → timelockGuard.canSend()?
  → warmup.canSend()?
  → contactGraph.canMessage()?
  → replyRatioGuard.beforeSend()?
  → reconnectThrottle.beforeSend()?
  → rateLimiter.getDelay() [with group-scaled limits if applicable]
  → presenceChoreographer multipliers
  → return { allowed, delayMs, health }

afterSend(jid, text)
  → record in all modules
  → stateManager.save() [debounced 5s]

afterHealthEvent() [called internally on ban/restriction events]
  → stateManager.save() [immediate, not debounced — ban events must persist]
```

## Migration Guide (v2 → v3)

Compat shim handles most cases automatically. Only change needed:

```ts
// v2 (still works, logs deprecation warning)
new AntiBan({ rateLimiter: { maxPerMinute: 8 } })

// v3 (preferred)
new AntiBan({ maxPerMinute: 8 })
```

Advanced v2 configs (sessionStability, jidCanonicalizer) still pass-through as-is.

## Out of Scope (v4)

- Proxy-per-session
- Multi-number pool manager (explicit v4 — pool N sessions under one AntiBan coordinator, round-robin with per-number warmup state)
- Per-group rate overrides (`groupOverrides: Record<jid, Partial<RateLimitConfig>>`)
- Newsletter JID separate profile (currently aliases to group behavior)
- File lock for concurrent state.json writers
- Remove compat shim
